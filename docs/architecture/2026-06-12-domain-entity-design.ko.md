# Echo 도메인 엔티티 설계

## 상태

이 문서는 데이터베이스 테이블을 설계하기 전에 합의한 도메인 엔티티 모델을 정의한다.
데이터베이스 스키마 문서가 아니며, 테이블 구조, 제약 조건, RLS, 인덱스는 이 모델을 검토한 뒤 도출해야 한다.

## 연습 유형

```text
연습 유형 =
  롤플레잉 연습
  OR 암기 연습
```

MVP는 롤플레잉과 암기, 두 가지 연습 유형만 지원한다.

## 사용자

```text
인증 사용자 =
  Supabase Auth 사용자

Echo 사용자 프로필 =
  인증 사용자와 1:1로 연결되는 Echo 내부 사용자 엔티티
  AND 선택적 표시 이름
  AND 생성 시각
  AND 수정 시각
```

규칙:

- `표시 이름`은 선택이다.
- `표시 이름`이 비어 있으면 앱은 생성된 fallback/random 이름을 보여줄 수 있다.
- fallback 이름은 화면 표시용이며, 사용자가 저장하기 전까지 프로필 값으로 저장하지 않는다.
- 추후 사용자 설정은 `Echo 사용자 프로필`을 확장한다.

## 자료

```text
자료 =
  반복 사용 가능한 연습 템플릿

자료 상태 =
  활성
  OR 삭제됨
```

규칙:

- 활성 자료는 자료 목록에 표시되며 새 세션을 만들 수 있다.
- 삭제된 자료는 자료 목록에서 숨겨지고 새 세션을 만들 수 없다.
- 자료를 삭제해도 기존 세션이나 세션 스냅샷은 삭제하지 않는다.

## 태그

```text
태그 값 =
  표시 이름
  AND 정규화 이름

자료 태그 =
  순서가 없는 태그 값 집합

세션 스냅샷 태그 =
  세션 시작 시점에 복사된 자료 태그
```

규칙:

- 태그는 사용자가 자료를 정리하고 자료/세션 목록을 필터링하기 위해 관리하는 라벨이다.
- 태그에는 순서가 없다. 도메인에는 태그 순서가 존재하지 않는다.
- 태그 이름은 비어 있을 수 없다.
- 앞뒤 공백은 제거한다.
- 중복 판정은 `정규화 이름`을 기준으로 한다.
- 태그 비교와 필터링은 대소문자를 구분하지 않는다.
- UI 표시는 `표시 이름`을 사용한다.
- 세션 스냅샷 태그는 세션이 시작된 시점의 자료 태그를 보존한다.

## 롤플레잉 자료

```text
롤플레잉 자료 =
  소유자
  AND 제목
  AND 상황/맥락
  AND 태그
  AND 정확히 2명의 롤플레잉 화자
  AND 롤플레잉 대사 목록
  AND 자료 상태
```

```text
롤플레잉 화자 =
  표시 이름

롤플레잉 대사 =
  대사 순서
  AND 화자
  AND 텍스트
  AND 선택적 번역
```

규칙:

- 롤플레잉 자료는 정확히 두 명의 화자를 가진다.
- 모든 롤플레잉 대사는 두 화자 중 한 명을 참조한다.
- `텍스트`는 필수다.
- `번역`은 선택이다.
- 난이도는 자료의 본질적인 속성이 아니므로 MVP에서 제외한다.

## 암기 자료

```text
암기 자료 =
  소유자
  AND 제목
  AND 태그
  AND 암기 문단 목록
  AND 자료 상태
```

```text
암기 문단 =
  문단 순서
  AND 암기 문장 목록

암기 문장 =
  문장 순서
  AND 텍스트
  AND 선택적 번역
```

규칙:

- 암기 자료는 적어도 하나의 문단을 가진다.
- 각 문단은 적어도 하나의 문장을 가진다.
- 각 문장은 `텍스트`를 필수로 가진다.
- `번역`은 선택이다.
- MVP 암기 세션은 전체 자료를 문단 순서와 문장 순서대로 사용한다.
- 이후 버전에서는 문단 또는 문장 범위 기반의 부분 세션을 지원할 수 있다.

## 세션

```text
세션 =
  자료 기반 실행 기록
  AND 세션 시작 시점 스냅샷
  AND 진행 상태

세션 상태 =
  준비됨
  OR 진행 중
  OR 완료됨
  OR 삭제됨
```

규칙:

- 세션은 원본 자료와 분리된다.
- 세션은 현재 자료가 아니라 세션 스냅샷을 기준으로 실행된다.
- 자료 수정이나 삭제는 기존 세션 스냅샷을 변경하지 않는다.
- 브라우저 종료는 MVP에서 `중단됨` 상태를 만들지 않는다. 완료되지 않은 세션은 `진행 중`으로 남는다.
- 삭제된 세션은 스냅샷, draft 녹음, accepted 녹음, 분석 작업, target 분석 결과, 세션 요약을 제거하거나 정리한다.

## 롤플레잉 세션

```text
롤플레잉 세션 =
  롤플레잉 자료에서 생성된 세션
  AND 선택된 학습자 화자
  AND 롤플레잉 세션 스냅샷
  AND 세션 상태
```

```text
롤플레잉 세션 스냅샷 =
  자료 제목 스냅샷
  AND 상황/맥락 스냅샷
  AND 태그 스냅샷
  AND 정확히 2명의 화자 스냅샷
  AND 대사 스냅샷 목록
```

규칙:

- 하나의 롤플레잉 세션은 정확히 하나의 학습자 화자를 가진다.
- 다른 화자를 연습하려면 사용자는 다른 세션을 생성한다.
- 대사의 화자가 선택된 학습자 화자이면 사용자가 해당 대사를 녹음한다.
- 대사의 화자가 선택된 학습자 화자가 아니면 AI 또는 앱이 상대 대사를 제시한다.

## 암기 세션

```text
암기 세션 =
  암기 자료에서 생성된 세션
  AND 암기 세션 스냅샷
  AND 세션 상태
```

```text
암기 세션 스냅샷 =
  자료 제목 스냅샷
  AND 태그 스냅샷
  AND 문단 스냅샷 목록
  AND 문장 스냅샷 목록
```

규칙:

- 암기 세션 흐름은 문단 단위로 진입한다.
- 녹음과 target 단위 분석은 문장 스냅샷을 기준으로 한다.
- MVP 세션은 모든 문단과 문장을 순서대로 다룬다.

## 세션 스냅샷 생명주기

```text
세션 스냅샷 =
  세션 시작 시점의 자료 내용 복사본
```

규칙:

- 세션 스냅샷은 세션과 함께 생성된다.
- 세션 스냅샷은 세션이 존재하는 동안 보존된다.
- 세션 스냅샷은 세션과 함께 삭제된다.
- 자료 삭제는 세션 스냅샷을 삭제하지 않는다.
- 스냅샷 저장량은 보존된 세션 수에 따라 증가하지만, 텍스트 스냅샷은 오디오 녹음 저장량에 비해 작다.

## 연습 대상

```text
연습 대상 =
  롤플레잉 대사 스냅샷
  OR 암기 문장 스냅샷
```

규칙:

- 녹음과 target 단위 분석 결과는 연습 대상에 연결된다.
- 연습 대상은 현재 자료가 아니라 세션 스냅샷에 속한다.

## 녹음

```text
녹음 오디오 =
  storage bucket
  AND object path
  AND mime type
  AND size
  AND duration
```

```text
Draft 녹음 =
  세션
  AND 연습 대상
  AND 녹음 오디오

Accepted 녹음 =
  세션
  AND 연습 대상
  AND 녹음 오디오
```

규칙:

- `녹음 오디오`는 MVP에서 독립 엔티티가 아니라 녹음이 소유하는 값 객체다.
- `Draft 녹음`은 데이터베이스에 저장된다.
- `Draft 녹음`은 세션 결과가 아니며 분석하지 않는다.
- 하나의 연습 대상은 0개 또는 1개의 draft 녹음을 가진다.
- 하나의 연습 대상은 0개 또는 1개의 accepted 녹음을 가진다.
- 새 draft를 만들면 이전 draft는 폐기되고 해당 오디오 cleanup을 예약한다.
- draft를 accept하면 accepted 녹음을 생성하거나 교체한다.
- 교체된 accepted 녹음은 보존하지 않는다.
- 사용자가 draft를 accept하기 전에 나가면 draft는 복원하지 않고 cleanup 대상이 된다.
- 추후 no-script 회화나 독립 오디오 처리 기능이 생기면 `녹음 오디오`를 `오디오 파일` 엔티티로 승격할 수 있다.

## 분석

```text
분석 작업 =
  완료된 세션에 대해 생성되는 batch 분석 요청

연습 대상 분석 결과 =
  하나의 연습 대상에 대한 분석 결과

세션 분석 요약 =
  세션 전체에 대한 요약 분석 결과
```

규칙:

- MVP는 세션 진행 중에는 분석하지 않는다.
- 세션을 완료하면 batch 분석 작업을 queue에 넣는다.
- 분석은 accepted 녹음만 사용한다.
- 결과는 target 단위 결과와 세션 단위 요약을 포함한다.
- 이후 실시간 피드백은 accepted 녹음을 더 이른 시점에 분석하는 방식으로 추가할 수 있으며, target 모델을 바꾸지 않아도 된다.

## Cleanup

```text
Cleanup 실패 로그 =
  storage cleanup 실패 기록
```

규칙:

- Cleanup 실패 로그는 학습 결과가 아니라 운영 보조 기록이다.
- Draft 녹음 cleanup 실패는 cleanup 실패 로그를 만든다.
- Accepted 녹음 교체 또는 삭제 실패는 cleanup 실패 로그를 만든다.
- 세션 삭제 cleanup 실패는 cleanup 실패 로그를 만든다.
- Cleanup 실패 로그는 추후 Datadog, Sentry, metric, retry 연동의 기반이 된다.

## 도출된 설계 방향

합의된 모델은 도메인 명시적 설계다.

- 자료와 세션은 별도 엔티티다.
- 세션 스냅샷 target은 명시적으로 둔다.
- Draft 녹음과 Accepted 녹음은 별도 엔티티다.
- 오디오 메타데이터는 MVP에서 녹음이 소유하는 값 객체로 남긴다.
- 분석은 세션 완료 후 batch 기반이며, target 단위 결과와 세션 단위 결과를 가진다.

기각된 대안:

- 넓은 JSON 스냅샷만 사용하는 DB 단순 모델은 기각한다. 녹음과 분석이 안정적인 target 참조를 필요로 하기 때문이다.
- 자료 versioning 모델은 MVP에 비해 version 관리 비용이 크므로 기각한다.
